## Problem Statement

VeasyGuide's research pipeline (activity detection on slide-based lecture videos: pointing, marking, sketching regions) and its accessibility player (highlight overlay + content-following magnification, validated in a user study) are locked inside a study rig. The analysis only runs as a hand-executed offline Python script; results are pre-committed JSON. Nobody outside the study can use it.

Goal: a public web app where anyone drops in a lecture video and watches it with the accessibility enhancements — analysis running fully in their browser, playback starting within seconds, no upload, no account, no server ever touching the video.

Primary users: **low-vision learners** (defaults tuned for them) and **educators** (activity timeline as a lens on their own content). The **whole webapp is the artifact** — not a library, not a paper demo.

## What Makes This Cool

Three "whoa"s, deliberately balanced:
1. **Instant magic** — drop a file, playback starts in seconds, highlights and zoom just appear while analysis races ahead in the background.
2. **Accessibility superpower** — a lecture video becomes followable: magnification tracks the instructor's pen, highlights guide the eye, everything tunable.
3. **X-ray vision** — an inspector surface (activity timeline, per-activity detail) shows what the machine found.

EUREKA (logged): conventional wisdom says content-aware video accessibility needs ML models and therefore servers (SViM uses a deep saliency net; MAVP uses multimodal LLMs). But for slide-based lecture video, **saliency ≡ change** — frame differencing + graph clustering finds the activity with no model. The "no ML" constraint is the feature: it runs anywhere, private, free, client-side.

## Constraints

- **No backend in v1.** Static site. Video never leaves the device. No video storage anywhere.
- **No YouTube ingestion in v1** (ToS + CORS; rejected deliberately). Future compliant option noted: getDisplayMedia tab capture of the official embed.
- **Target hardware is weak.** Low-vision learners skew toward school-issued Chromebooks and older laptops, often already running screen magnifiers. Performance claims must be measured, not assumed.
- **Activity TYPE classification is out of v1** (pinned; future idea: LLM classification on sampled frames per activity). The v1 inspector and timeline are therefore type-less (v1 encoding: single accent color for valid activities, width encodes duration, invalid hidden by default).
- **Browsers:** v1 ships Chromium-only, enforced by a capability check on load (WebCodecs `VideoDecoder` presence) with a clear notice on other browsers. A `<video>`-seek + canvas fallback decoder for Firefox/Safari is scheduled for v1.x, not v1. All v1 success criteria are scoped to Chromium.
- **Codecs:** supported input is what the user's hardware can decode via WebCodecs — H.264/VP9/AV1 in MP4/WebM/MKV containers in practice. On drop, probe the actual track with `VideoDecoder.isConfigSupported()`; unsupported input (e.g., HEVC on most Chromebooks) fails immediately with a user-facing message naming the codec, not a hang.
- Deploy target: Google Cloud (static hosting).

## Premises (final, as revised after second opinion)

1. **Pure-TypeScript pipeline, no OpenCV.js.** With type classification cut, `cv2.matchShapes` has no consumer (verified: `difference` only feeds `_extract_components` and `_calc_type` in `roi.py`; the frontend uses `components` only as a count). Remaining ops — absdiff, blur, threshold, dilate, connected-component boxes — are ~120 lines of plain TS on raw pixel buffers. Demux/decode via **Mediabunny** (WebCodecs; handles container/codec edge cases).
2. **Faithful port first, optimize second.** Port with the study parameters (`sample_fps_ratio=0.2`, `span_th=1`) and WITHOUT node merging (the Python merge loop `analyzer.py:161` is dead code — `while not keep_merging` never runs — so committed outputs have no contraction). Validate against the committed Python JSONs (golden files) at native resolution within the tolerances defined in Success Criteria. Only then enable the ~480p downscale fast path (thresholds are relative fractions and survive scaling; fixed 5×5 blur / 3× dilate kernels act relatively larger at 480p) — the downscale path must meet the **identical** tolerances against the same golden files. If it breaks tolerance (plausible — the fixed kernels act relatively larger): first scale kernel sizes proportionally with resolution and re-validate; if still failing, the downscale path is dropped and the 2–<4× mitigation band re-gates on lower sample rate alone.
3. **Streaming by design.** The edge criterion is temporally local — edges connect nodes within `span_th` (1s in v1, the study parameter; the code default 1.5s is only the upper design bound for the window) — so components finalize once the analysis frontier passes `span_th` beyond their last node (watermark). Playback gates on a **10 s analyzed lead** (fixed; same value referenced in Recommended Approach). If playback catches the frontier mid-play (e.g., slow segment restart after a seek), the player pauses with an "analysis catching up" state and auto-resumes once the lead is restored — it never plays un-enhanced within an unanalyzed range. Sampling is **time-based** (media timestamps, every `sample_fps_ratio` seconds), not frame-index-based, so variable-frame-rate input (common in screen-recorded lectures) degrades gracefully; VFR is a known input class for validation. **Segment lifecycle on seek:** seeking past the frontier aborts the current segment's decode, starts a new segment at the seek point (treated as a scene start), and records both ranges; when a segment's frontier reaches an already-analyzed range it stops; when idle, the worker resumes the earliest unanalyzed gap. No cross-segment stitching in v1 (worst case: one activity briefly appears as two at a seek boundary — cosmetic, accepted). Scene detection stays (player resets zoom/activity on scene change); a simple global-diff cut detector on already-sampled frames replaces PySceneDetect.
4. **Performance is measured, not assumed.** A ×-realtime HUD exists from day one and ships (as the subtle "analyzing N× realtime" meter). A Phase 0 throughput spike on weak hardware — measuring analysis decode + playback decode + a magnification-rate canvas blit (stand-in for `MagnificationOverlay`'s per-frame redraw) running simultaneously, including a seek into unanalyzed territory (seeks pay a keyframe penalty: decode restarts from the previous keyframe, and lecture GOPs can be multi-second) — gates the full buildout. Disjoint decision bands on the low-end device: **≥4× realtime = pass as designed; 2–<4× = ship with mitigations** (downscale analysis path, lower sample rate); **<2× = accepted degraded mode**: the app offers "analyze first, then play" with an upfront time estimate (~3 min for a 6-min video at 2×) — explicitly accepted as the worst case rather than adding a server tier, which stays out of v1 scope. **Runtime trigger for degraded mode:** the shipped app measures ×-realtime over the first 10 s of analysis (the same measurement the HUD displays, so Phase 0 numbers and runtime decisions are directly comparable; excludes the initial keyframe warm-up burst); if the projection is <2×, it offers analyze-first mode and remembers the choice per device in localStorage.
5. **Client-side caching.** Only **complete** analyses are cached (a tab closed mid-analysis re-analyzes next time; persisting partial segment ranges is v1.x at most). Cached in IndexedDB keyed by a content fingerprint: **SHA-256 over (first 2 MB ∥ last 2 MB ∥ file size ∥ container duration rounded to integer milliseconds)** — never a full-file hash, and no floats in the key. Rewatching the same file loads instantly. LRU-capped at 100 entries; quota errors degrade gracefully (skip caching, never block playback); a "clear cached analyses" affordance lives in settings. Manual export/import of the analysis JSON for sharing/backup; the export IS the cached reduced payload plus a `schemaVersion` field; on import, if the JSON's fingerprint doesn't match the loaded video, warn and allow use with a persistent "analysis/video mismatch" badge. The reduced payload drops `components[].nodes` (~95% of size; only the count is used) — ~1.4 MB → ~50 KB per video.
6. **The player migrates nearly as-is.** `VideoPlayer`, `HighlightIndicator`, `MagnificationOverlay` (keeping its proven per-frame `<video>`→canvas redraw in v1), `IndicatorPointer`, `SVGFilters`, both settings suites + Zustand stores, `ActivityTimeline`/`ActivityInfo` (type-less variants). All study scaffolding dropped: modules, tasks, Firestore, `logAction`, admin pages, basic auth. Output schema keeps `is_valid`, `metadata.duration`, `pos/dim`, `start/end` (the player filters on them).

## Approaches Considered

### Approach A: Single App, Spike-First (CHOSEN)
One Vite + React + TS repo, one website, one deploy. The analyzer lives in `src/analyzer/` with **zero React imports** and golden-file tests, so it stays extractable if a library consumer ever materializes. Phase 0 is the throughput spike wearing the demo's clothes.
- Effort: M (human ~2–3 weeks / CC ~2–4 sessions) · Risk: Low
- Pros: fastest to the actual artifact (the website); fewest moving parts; no speculative structure
- Cons: publishing the analyzer as a package later requires (an afternoon of) extraction

### Approach B: Monorepo — Analyzer as a Library
Same single repo/site, internally split `packages/analyzer` + `apps/web` via npm workspaces.
- Effort: M–L · Risk: Low–Med
- Rejected because: the user clarified "the webapp is the artifact, the whole thing is the artifact" — workspace structure would serve an npm consumer that doesn't exist. A's framework-free folder + golden tests preserves the option.

### Approach C: The Live Lens (creative/lateral, from second opinion)
getDisplayMedia capture → same pipeline in true realtime → PiP overlay following the pen in live lectures.
- Effort: L–XL · Risk: High
- Rejected for v1: abandons validated ground (recorded playback), realtime constraint hits weakest hardware hardest, PiP compositing fragile. **Named v2 direction.**

## Recommended Approach

**A — Single App, Spike-First.** Architecture:

```
new-repo/
├── docs/                    # wireframe-v1.html/.png (copied from design doc dir)
├── src/
│   ├── analyzer/            # framework-free TS (no React imports)
│   │   ├── demux.ts         # Mediabunny wrapper: sequential decode from t, sampled frames
│   │   ├── pipeline.ts      # grayscale absdiff → blur → threshold → dilate → blob boxes (~120 lines)
│   │   ├── scenes.ts        # global-diff cut detector (~20 lines)
│   │   ├── graph.ts         # time-sorted windowed union-find; watermark finalizer
│   │   ├── segments.ts      # processed-range tracking; seek → new segment
│   │   ├── worker.ts        # Web Worker entry; postMessage protocol (activity/scene/progress events)
│   │   └── types.ts         # output schema (compatible with existing VideoAnalysis types)
│   ├── components/          # ported: VideoPlayer, HighlightIndicator, MagnificationOverlay,
│   │   ...                  #   settings suites, ActivityTimeline (type-less), SVGFilters
│   ├── stores/              # ported Zustand stores + new AnalysisStore (streaming append)
│   ├── cache/               # IndexedDB (fingerprint → analysis JSON), export/import
│   └── pages/               # Landing (drop zone) / Player (per approved wireframe)
├── golden/                  # Python output JSONs + validation runner
└── .github/workflows/       # CI: test + build + deploy on merge
```

Key runtime flow: drop file → codec probe (`VideoDecoder.isConfigSupported`; friendly failure for unsupported codecs) → fingerprint → cache hit? load : spawn worker → worker decodes sequentially, emits scenes + finalized activities + frontier position → store appends → player gates playback on a **10 s analyzed lead** (fixed) → seek past frontier aborts current segment and starts a new one (idle worker backfills earliest gap) → HUD/meter shows ×-realtime + % done. **Mid-decode errors** (corrupt/truncated streams): keep all finalized results, mark the rest of the range unanalyzed, show a non-blocking notice; playback continues without enhancements past the failure point. Seek bar renders played + analyzed-frontier shading; activity strip beneath. Approved wireframe: `~/.gstack/projects/tomfluff-VeasyGuide/designs/wireframe-v1.html` (+ `.png`) — two states: landing with drop-zone hero + privacy promise, player with frontier shading and meter. ## Success Criteria

(All criteria scoped to Chromium browsers — see Constraints.)

- **Phase 0 gate** (6-min 720p H.264 lecture on the low-end benchmark device; analysis decode + playback decode + magnification-rate canvas blit running simultaneously, including one seek into unanalyzed territory): **≥4× realtime** and playback starts <15s wall-clock with no dropped playback frames (measured via `video.getVideoPlaybackQuality().droppedVideoFrames`) = pass as designed; **2–<4×** = proceed with mitigations (downscale, lower sample rate) and re-measure; **<2×** = degraded "analyze first" mode is the accepted floor (upfront time estimate shown; ~3 min wait for a 6-min video at 2×) — decide go/no-go consciously, not ad hoc.
- **Port validation:** activities from the TS pipeline match the committed Python JSONs — or regenerated goldens per the drift fallback below — on ≥ 2 study videos (same parameters, native res, no merging). **Matching procedure for the golden runner:** pair activities greedily by highest spatiotemporal IoU (temporal-interval IoU × bbox IoU), one-to-one; every matched pair must have bbox IoU ≥ 0.8 and start/end within ±0.5s; unmatched activities must satisfy `unmatched ≤ ceil(0.10 × golden_count)` per side, per video, else fail. Scene cuts within ±0.5s of golden scene boundaries, with no more than 1 spurious/missed cut per video. **Validate sampled-frame timestamp alignment first** — WebCodecs and ffmpeg/OpenCV may select different frames for the same nominal timestamps, and that (not pipeline bugs) is the likely first failure mode; if decoder drift alone breaks tolerance, regenerate goldens through a reference harness on the TS decode path and validate the pipeline against those (this validates pipeline math but no longer frame-selection fidelity — an accepted narrowing, noted here so "v1 done" is unambiguous about which goldens count).
- **CI vs local validation:** study videos are not committed to the repo (redistribution unclear); full golden validation is a local script pointing at local video files. CI commits one short self-owned clip — `Chem30secExmaple.mp4`, the user's own 30 s recording already used in the study repo (golden: `backend/data/analysis/Chem30secExmaple_output.json`) — for an end-to-end decode+pipeline test on every PR; the pipeline stages additionally get pure unit tests on small committed frame fixtures.
- **v1 done:** drop a local lecture video in a supported codec (H.264/VP9/AV1; unsupported codecs fail with a clear message) → enhanced playback with highlight + magnification + settings suites + type-less timeline; frontier shading on the seek bar; seek anywhere works (segments); re-drop of same file loads from cache instantly; export/import JSON round-trips (mismatched import warns + badges); deployed at a public URL; README documents setup + deploy.
